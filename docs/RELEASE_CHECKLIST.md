# Release Checklist

Run commands from the repository root.  Use the pinned Lean toolchain in
`lean-toolchain` and the committed dependency revisions in `lake-manifest.json`.

Do not run `lake clean`, `lake clear`, or delete `.lake/` as a cleanup step.

## Reviewer Path

This is the fast path intended for artifact reviewers and ordinary reproducible
checks.  It uses native proof mode for expensive finite proof leaves.

```bash
scripts/build-proof-mode.sh native covering_codes
lake -KproofMode=native exe covering_codes 3 6 1
lake -KproofMode=native exe covering_codes 3 8 3
lake -KproofMode=native exe covering_codes 2 7 1
scripts/check-generated-metadata.sh
python3 -B reference-data/scripts/check_reference_keys.py
python3 -B reference-data/scripts/compare_lean_post_keri_bounds.py
```

If the machine has many CPU cores but limited memory per worker, set a
conservative thread count first, for example:

```bash
export LEAN_NUM_THREADS=4
```

## Developer Regeneration Path

Run this after changes to primitive sources, derived sources, closure rules,
trace serialization, or the table generator.  This path regenerates the
committed table in native proof mode and verifies that generated files are up to
date.

```bash
scripts/check-generated.sh
```

`reference-data/reference-db-comparison.csv` is generated by the reference-data
comparison script and should remain uncommitted unless intentionally promoted.

## Release Path

Run the release path before creating the AFM submission tag.  Unlike the
reviewer path, release validation uses kernel proof mode.

```bash
scripts/check.sh
lake -KproofMode=kernel exe covering_codes 3 6 1
scripts/check-generated-metadata.sh
python3 -B reference-data/scripts/check_reference_keys.py
python3 -B reference-data/scripts/compare_lean_post_keri_bounds.py
```

If a full generated-table regeneration is required for the release, run the
developer regeneration path first, inspect the generated diff, and then rerun
the kernel release path.

Optional high-memory kernel checks, especially van Laarhoven explicit-code
certificates, should be run only with the opt-in and memory guidance documented
in `README.md`.

## Repository Hygiene

Before the final commit and tag, check that no accidental archives, logs, PDFs,
or large private files are present:

```bash
find . -type f -size +25M -not -path './.git/*' -not -path './.lake/*' -print
find . -type f \( -name '*.tar' -o -name '*.tar.gz' -o -name '*.zip' -o -name '*.pdf' -o -name '*.log' \) -print
```

If `gitleaks` is available, also run:

```bash
gitleaks detect --no-git --source .
```

Finally inspect the Git state:

```bash
git status --short
git status --ignored --short
```

## Tag And Archive

After all required release checks pass, create an annotated release tag.  Use a
stable, descriptive tag name, for example `afm-2026-submission-v1` for the
first AFM submission artifact.

```bash
release_tag="<release-tag>"
git tag -a "${release_tag}" -m "<release description>"
git push origin "${release_tag}"
git rev-parse "${release_tag}"
```

Do not move or rewrite a tag after the paper cites it.  If the source artifact
changes after citation, create a new tag instead of mutating the old one.

After the public repository is available, request Software Heritage archival and
record the SWHID when available.

## Paper Records

Record these values for the manuscript and release notes:

- repository URL;
- AFM submission tag;
- full commit hash;
- Software Heritage SWHID, once available.
