# Reference Data

This directory contains machine-readable reference data used for historical
comparison and audit work.  These files are not imported by the Lean proof
database and do not prove or certify any covering-code bound.

Certified bounds live in the Lean sources under `CoveringCodes/Database/` and
in the generated proof-carrying table.  Reference CSV files in this directory
are inputs to comparison scripts only.

`references.bib` is the shared BibTeX key registry for normalized reference
tables and Lean primitive provenance labels.  Keys use these prefixes:

- `lit_`: external literature.
- `lean_`: repository-local Lean source.
- `rule_`: algorithmic, structural, or elementary rule.

## Keri

Historical Keri web source: <https://old.sztaki.hu/~keri/codes/index.htm>.

`keri/non_mixed_covering_codes.csv` is a machine-readable transcription of the
Keri non-mixed covering-code tables.  It is treated as the historical Keri table
snapshot whose web page was last updated on 2011-11-21.

The CSV records table data, OCR/transcription details, and source references in
its own columns.  No separate BibTeX file is maintained for this snapshot.  OCR
or transcription corrections are documented in the CSV notes column.

## Post-Keri

`post-keri/non_mixed_covering_codes.csv` is the extendable reference table used
by `reference-data/scripts/compare_lean_post_keri_bounds.py`.  This release was
converted from the Keri snapshot; later literature updates should be applied here,
not to the historical Keri transcription.

Bibliographic entries for this table are collected in `references.bib`.
Entries derived from Keri source labels can be refined as the corresponding
references are audited.

The post-Keri CSV shape is intentionally smaller:

```text
q,n,r,lower_bound,upper_bound,lower_bound_reference,upper_bound_reference
```

The reference columns should contain BibTeX keys from
`references.bib`.  Multiple keys may be separated by semicolons.

`lean-post-keri-comparison-history.csv` is an append-only audit ledger for the
aggregate summary printed by `reference-data/scripts/compare_lean_post_keri_bounds.py`.
Each row compares the Lean generated table with the post-Keri reference table
over overlapping `(q,n,r)` triples that have both reference bounds.  The first
column is the commit timestamp in UTC, followed by the total and tag counts,
then a short commit hash and an optional `improvement` note.  Percentages are
derivable from `total` and are not stored.

The compact tag columns mean: `ref_tight` = `reference_tighter`, `exact` =
`exact_agree`, `ref_lo` = `reference_better_lower`, `ref_hi` =
`reference_better_upper`, `we_lo` = `we_better_lower`, `we_tight` =
`we_tighter`, `we_hi` = `we_better_upper`, `agree_lo` = `agree_lower`,
`agree_hi` = `agree_upper`, `missing` = `lean_missing`, and `contrad` =
contradictions.

The `improvement` column is intentionally sparse.  It records only bounds whose
comparison tag changed relative to the previous history row, using compact forms
such as `K_7(4,2)<=23 (was 25)` or `K_8(4,2)=23 (was [13,32])`; derived bound
changes that do not move an aggregate tag are omitted.

To update the history ledger, run the comparison script from the repository root,
copy the printed `COMPARISON SUMMARY` counts into a new row, record the current
commit hash, and leave absent tags as `0`.  If the row changes comparison tags,
add the corresponding short bound changes in `improvement`.  This file is not
updated automatically; it is intended for occasional human or agent-driven
snapshots.
The detailed row-level report remains `reference-data/reference-db-comparison.csv`
and is still generated/ignored.

## Lean Dump

`lean/non_mixed_covering_codes.csv` is a documentation dump of the generated
Lean table for `q >= 2`, `n >= 1`, and `r >= 1`, using the same normalized CSV
shape as the post-Keri table.  It is generated by a Lean/Lake executable that
imports the compiled proof-carrying table.

Each reference cell contains the full proof-trace provenance for that bound as
semicolon-separated BibTeX keys with no spaces around semicolons.  Primitive
trace labels are included directly, and derived trace steps add `rule_*` keys.

Regenerate it with:

```bash
lake -KproofMode=native exe reference_data_dump
```

## Consistency Checks

Run:

```bash
python3 -B reference-data/scripts/check_reference_keys.py
```

This checks that normalized CSV reference columns and Lean primitive provenance
keys all resolve to entries in `references.bib`, and that BibTeX keys use one
of the documented prefixes.
